Stackr

A declarative, composable homelab deployment tool. Define your self-hosted stack in a single stackr.yml file — Stackr renders Jinja2 Docker Compose templates, validates your config, and deploys everything with a single command.

Python 3.11+ Docker Compose nginx-proxy-manager default 50+ catalog apps

Overview

Stackr manages your entire homelab as code. Instead of maintaining individual docker-compose.yml files per app, you maintain one stackr.yml that declares which apps are enabled and how they should be configured. Stackr handles the rest.

How it works

  1. You run stackr init to generate a stackr.yml from an interactive wizard.
  2. You toggle apps on or off via stackr ui (TUI) or by editing the YAML directly.
  3. Stackr validates your config, renders Jinja2 templates for each enabled app, and runs docker compose up -d.
  4. State is tracked in ~/.stackr/state.json — unchanged apps are skipped on the next deploy.

Key concepts

ConceptDescription
App catalogA library of pre-built app.yml + compose.yml.j2 pairs for 50+ popular apps.
stackr.ymlYour personal config — global settings, network mode, enabled apps, vars, and secrets references.
.stackr.envSecrets file (gitignored). Auto-generated; holds API tokens, passwords, and auto-generated secrets.
State file~/.stackr/state.json — tracks deployed compose hashes and image digests for drift detection.
Reverse proxynginx-proxy-manager is the sole reverse proxy. Apps join the proxy Docker network; routing is configured through NPM's web UI by pointing to container names.

Installation

Requirements

  • Python 3.11 or later
  • Docker Engine with Compose v2 (docker compose — not docker-compose)
  • pipx (recommended) or pip

Install via the bootstrap script (recommended)

curl -fsSL https://raw.githubusercontent.com/your-org/stackr/main/install.sh | bash

The installer bootstraps pipx if it is not present, installs Stackr into an isolated environment, and adds it to your PATH.

Install with pipx

# Install pipx if you don't have it
python3 -m pip install --user pipx
python3 -m pipx ensurepath

# Install Stackr
pipx install git+https://github.com/your-org/stackr.git

Verify the installation

stackr --version
# stackr 0.4.2
Tip
If stackr is not found after install, reload your shell: exec $SHELL — the installer writes ~/.local/bin to your shell rc file, but the current session needs to pick it up.

Quick Start

Get from zero to a running stack in under five minutes.

  1. Run the setup wizard 
    stackr init
    Answer a few prompts (data directory, domain, timezone). Stackr writes stackr.yml and .stackr.env in the current directory. nginx-proxy-manager is deployed as the reverse proxy — configure routing rules through its web UI after deploy.
  2. Add any secrets to .stackr.env 
    # Database password (can also be auto-generated by Stackr)
POSTGRES_PASSWORD=super-secret
  3. Toggle apps on/off interactively 
    stackr ui
    A terminal UI shows all catalog apps. Press Space to toggle, Enter to confirm, q to quit and save.
  4. Validate your configuration 
    stackr validate
    Stackr checks port conflicts, missing secrets, dependency order, and more.
  5. Deploy everything 
    stackr deploy
    Stackr pulls images, renders compose files, and starts every enabled app.
  6. Check status 
    stackr status

stackr.yml Reference

The config file lives in your project directory (wherever you run Stackr commands from). Use --config / -c on any command to point to a different file.

global

global:
  data_dir: /opt/appdata    # Where container volumes are stored on the host
  timezone: America/New_York
  puid: 1000             # UID for container processes
  pgid: 1000             # GID for container processes

network

network:
  domain: example.com    # Your domain — used as a reference in templates and the web UI
  local_domain: home.example.com  # Internal-only domain (optional)
Reverse proxy
Stackr uses nginx-proxy-manager as the sole reverse proxy. Apps join the proxy Docker network. After deploying, open NPM's web UI and create proxy hosts pointing to each container name (e.g. http://jellyfin:8096). No label configuration or DNS provider credentials are required.

security

security:
  socket_proxy: true   # Proxy Docker socket access via tecnativa/docker-socket-proxy
  crowdsec: false      # Enable CrowdSec intrusion prevention (requires crowdsec app)

backup

backup:
  enabled: false
  destination: /mnt/backup  # Restic repository — local path or s3:bucket/prefix
  schedule: "0 2 * * *"     # Informational cron string (no built-in scheduler)

alerts

alerts:
  enabled: false
  provider: ntfy           # ntfy | gotify | webhook
  url: https://ntfy.sh/my-homelab
  token: ${NTFY_TOKEN}      # Optional Bearer token

apps

apps:
  # Minimal — just flip enabled: true
  - name: jellyfin
    enabled: true

  # With custom vars
  - name: qbittorrent
    enabled: true
    vars:
      webui_port: 8080

  # Local catalog path (custom app)
  - name: my-app
    enabled: true
    catalog_path: ./local-catalog/my-app

  # With raw compose overrides
  - name: postgres
    enabled: true
    overrides:
      services:
        postgres:
          mem_limit: 512m

Secrets & .stackr.env

Secrets are never stored directly in stackr.yml. Instead, reference them using the ${VAR_NAME} syntax. Stackr resolves them in this priority order:

  1. Shell environment — highest priority; set with export VAR=value
  2. .stackr.env file — in the same directory as stackr.yml
  3. Auto-generated — Stackr generates random values for passwords/keys on first deploy

.stackr.env file format

# DNS provider token
CF_DNS_API_TOKEN=my-cloudflare-token

# Database password (can also be auto-generated by Stackr)
POSTGRES_PASSWORD=super-secret

# Notification token
NTFY_TOKEN=bearer-token-here
Keep .stackr.env secret
This file is automatically added to .gitignore by stackr init. Never commit it to version control.

Security Options

Docker socket proxy

When security.socket_proxy: true, Stackr deploys docker-socket-proxy so apps that need Docker API access (e.g. Portainer, Dozzle) connect through it instead of mounting /var/run/docker.sock directly. This limits what a compromised container can do to your host.

Recommended for all deployments.

CrowdSec

Set security.crowdsec: true and add crowdsec to your enabled apps. CrowdSec acts as an intrusion-prevention system that reads NPM access logs and blocks malicious IPs. 

security:
  crowdsec: true

apps:
  - name: crowdsec
    enabled: true

Authentication — pocket-id + tinyauth

Stackr's security catalog provides two complementary apps for protecting your services:

AppDescription
pocket-id Lightweight OIDC provider with passkey-only authentication (no passwords). Acts as your identity server. Port 1411.
tinyauth Forward-auth proxy for apps that don't have native OIDC support. Sits in front of an app and delegates authentication to pocket-id. Requires pocket-id. Port 3000. 
apps:
  - name: pocket-id
    enabled: true

  - name: tinyauth
    enabled: true

Once deployed, configure NPM to use tinyauth as the forward-auth host for any apps you want to protect.

Alerts & Notifications

Stackr can push notifications on deploy failures and doctor errors. Configure the alerts block in stackr.yml: 

alerts:
  enabled: true
  provider: ntfy
  url: https://ntfy.sh/my-homelab-alerts
  token: ${NTFY_TOKEN}  # Optional

Supported providers: ntfy, Gotify, and any generic webhook (POST with JSON body).

Remote Mounts

Stackr can mount network shares before deploying apps, ensuring volumes are available. Configure under the mounts: key. 

mounts:
  - name: media
    type: smb            # smb | nfs | rclone
    remote: //192.168.1.10/media
    mountpoint: /mnt/media
    username: myuser
    password: ${SMB_PASSWORD}

  - name: photos
    type: nfs
    remote: 192.168.1.10:/export/photos
    mountpoint: /mnt/photos
    options: ro,noatime

Mount commands

stackr mount    # Mount all configured shares
stackr umount   # Unmount all configured shares
Requirements by type
smb — requires cifs-utils (apt install cifs-utils)
nfs — uses system mount
rclone — requires rclone binary with a configured remote

Updating the Catalog

Stackr ships a built-in catalog. You can update it independently of the main tool by downloading the latest release from GitHub. 

# Update to latest release
stackr catalog update

# Pin to a specific release tag
stackr catalog update --tag v1.5.0

# Show current catalog version and app count
stackr catalog version

The user catalog is installed at ~/.stackr/catalog/ and takes priority over the built-in catalog. To revert to the built-in catalog, delete that directory.

All-or-nothing overlay
If ~/.stackr/catalog/ contains any apps, it completely replaces the built-in catalog. You cannot partially overlay — your user catalog must contain all apps you want to use.

Validate & Plan

Validate

Check your config for errors before deploying:

stackr validate

Checks performed:

  • Required secrets are present in the environment
  • Port conflicts between apps
  • Hard dependencies (requires:) are enabled
  • DNS provider env vars are set
  • Security stack consistency (auth provider, CrowdSec)
  • Volume paths exist (warns for external: true volumes)

Plan (dry run)

See what would change without touching anything:

stackr plan

Compares the rendered compose output against the stored state. Shows deploy for apps that would be updated and no-op for unchanged ones.

Deploy

Deploy all enabled apps

stackr deploy

Stackr will:

  1. Run validation (abort if there are errors)
  2. Pull the latest images for each app
  3. Render the Jinja2 compose template
  4. Run docker compose up -d
  5. Record the state and image digests

Deploy a single app

stackr deploy jellyfin

Skip image pull

stackr deploy --skip-pull

Custom config file

stackr deploy --config /path/to/my-stack.yml

Update Images

Pull the latest images for all apps and redeploy only those with changes: 

stackr update

Unlike stackr deploy, stackr update skips an app only when both the compose content and all image digests are unchanged. This detects upstream image updates (e.g. :latest pointing to a new build) even when your config hasn't changed.

Tip
Schedule stackr update in a cron job or systemd timer to automatically keep your stack up to date: 
0 3 * * * cd /opt/stackr && stackr update >> /var/log/stackr-update.log 2>&1

Status

stackr status            # All apps
stackr status jellyfin   # Single app

Shows a table with each app's state:

StateMeaning
runningContainer is up and healthy.
stoppedContainer exists but is not running.
driftedRunning config differs from the last deployed state.
unknownApp has never been deployed or state is missing.

Stop & Restart

# Stop a running app
stackr stop jellyfin

# Restart without a full redeploy (just docker compose restart)
stackr restart jellyfin
Tip
stackr restart is a fast restart (equivalent to docker compose restart) and does not re-render templates or pull images. Use stackr deploy appname for a full redeploy of a single app.

Logs & Shell

Tail logs

# Follow logs (default)
stackr logs jellyfin

# Print without following
stackr logs jellyfin --no-follow

Open a shell inside a container

# Opens /bin/sh by default
stackr shell jellyfin

# Use bash
stackr shell jellyfin --shell bash

# Target a specific service in a multi-service app
stackr shell pocket-id --service pocket-id-db

Rollback

If an update breaks an app, roll back to the last known-good compose configuration: 

stackr rollback jellyfin

Stackr stores the full rendered compose YAML for each app in the state file. Rollback re-deploys that saved compose without pulling new images.

Note
Rollback restores the compose configuration, not the container data. Use stackr restore to restore data from a backup snapshot.

Remove an App

# Prompts for confirmation
stackr remove jellyfin

# Skip confirmation (for scripts)
stackr remove jellyfin --yes

This stops and removes the containers (docker compose down). Named volumes are not deleted — your data is safe. To also disable the app, set enabled: false in stackr.yml.

Doctor — Pre-flight Health Checks

stackr doctor

Runs a suite of pre-flight checks and reports pass/warn/fail for each:

  • Docker daemon is reachable
  • Required Docker networks exist (proxy, etc.)
  • Required secrets are present in the environment
  • Catalog is loadable and consistent
  • DNS provider env vars are set

Doctor is also called automatically before deploy. If any check fails, you get a push notification if alerts is configured.

Web UI

Stackr includes a browser-based dashboard powered by FastAPI and HTMX. 

# Start on localhost:8000 (foreground)
stackr web

# Listen on all interfaces, custom port
stackr web --host 0.0.0.0 --port 9000

The web UI provides:

  • Dashboard view of all apps with current status
  • Toggle apps on/off with a click (updates stackr.yml atomically)
  • Per-app and full-stack deploy buttons
  • Live log streaming (Server-Sent Events)
  • Full settings editor — tabbed panel covering every config section (see below)
  • Per-app var overrides — inline form with type-aware inputs

Per-app var overrides

App cards for catalog apps with configurable vars show a ⚙ Configure button. Click it to expand an inline form with type-aware inputs (select, boolean, integer, or text) derived from the catalog's VarDef definitions. Vars are saved per-app to stackr.yml.

Persistent Service

By default stackr web runs in the foreground and stops when the terminal closes. Use stackr service install to register it as a background service that starts automatically on login and survives reboots. 

# Install and start the service
stackr service install

# Custom host, port, and config file
stackr service install --host 0.0.0.0 --port 9000 --config /opt/stackr/stackr.yml
CommandDescription
stackr service installWrite unit/plist, enable, and start the service.
stackr service uninstallStop, disable, and delete the unit/plist.
stackr service startStart a stopped service.
stackr service stopStop the running service.
stackr service restartRestart the service (picks up config changes).
stackr service statusShow current status from the init system.

macOSlaunchd LaunchAgent~/Library/LaunchAgents/dev.stackr.web.plist
Tip — Linux lingering
For systemd user services to start at boot (before you log in), enable lingering once: 
loginctl enable-linger $USER

Backup & Restore

Stackr uses restic under the hood. You must have restic installed and accessible on PATH.

Run a backup now

stackr backup

Backs up data_dir, the state directory (~/.stackr/), and the config directory (where stackr.yml lives). The restic repository password is auto-generated on first run and stored in .stackr.env as STACKR_RESTIC_PASSWORD.

List snapshots

stackr snapshots

Restore from a snapshot

# Restore the latest snapshot to data_dir
stackr restore latest

# Restore a specific snapshot by ID
stackr restore a1b2c3d4

# Restore to a different directory
stackr restore latest --target /tmp/restore

Remote backup destinations

Set backup.destination to any restic-supported backend:

backup:
  enabled: true
  destination: s3:s3.amazonaws.com/my-bucket/stackr

Migrate from Deployrr

If you're coming from Deployrr, Stackr can generate a stackr.yml from your existing Deployrr app list: 

# Interactive — enter app names one per line
stackr migrate

# From a file (one app name per line)
stackr migrate --input deployrr-apps.txt

# Custom output path
stackr migrate --input deployrr-apps.txt --output my-stack.yml

Stackr fuzzy-matches Deployrr app names against the catalog and reports any names it couldn't map so you can add them manually.

Custom Local Apps

You can add apps not in the built-in catalog by pointing to a local directory that contains app.yml and compose.yml.j2. 

apps:
  - name: my-intranet
    enabled: true
    catalog_path: ./local-catalog/my-intranet

Minimal app.yml

name: my-intranet
display_name: My Intranet
description: Custom internal app
category: management
version: latest
exposure: internal
ports: [8080]
host_ports: []
volumes:
  - name: config
    path: /config

Minimal compose.yml.j2

services:
  my-intranet:
    image: myorg/my-intranet:{{ vars.version | default('latest') }}
    container_name: my-intranet
    restart: unless-stopped
    volumes:
      - {{ global.data_dir }}/my-intranet/config:/config
    networks:
      - proxy

networks:
  proxy:
    external: true

After deploying, add a proxy host in nginx-proxy-manager pointing to http://my-intranet:8080 with your desired domain.

Compose Overrides

Override any part of the rendered Docker Compose output without modifying the catalog template. Overrides are deep-merged on top of the rendered compose YAML. 

apps:
  - name: jellyfin
    enabled: true
    overrides:
      services:
        jellyfin:
          mem_limit: 4g
          devices:
            - /dev/dri:/dev/dri  # Pass through GPU for hardware transcoding

Render Debug

Print the fully rendered Docker Compose YAML for any app without deploying it. Useful for debugging template issues or verifying overrides: 

stackr render jellyfin
stackr render nginx-proxy-manager --config staging.yml

Upgrade & Uninstall

Upgrade to the latest version

stackr upgrade

Pulls the latest commit from the main GitHub branch and reinstalls the pipx package. After upgrading, run exec $SHELL if the version number looks unchanged.

Uninstall

# Interactive — prompts before deleting ~/.stackr
stackr uninstall

# Non-interactive — removes everything without prompts
stackr uninstall --yes

Removes the pipx package and optionally the ~/.stackr/ state directory. .stackr.env files in your project directories are left in place — delete them manually if you no longer need the secrets they contain.

File Locations (Linux)

All Stackr runtime files live under ~/.stackr/ and next to your stackr.yml. Nothing is written to system-wide paths except the data_dir you configure (default /opt/appdata).

PathWhat it is
stackr.yml Your config file. Lives wherever you ran stackr init (e.g. ~/stackr/stackr.yml). Pass --config to use a different path.
.stackr.env Secrets file — same directory as stackr.yml. Auto-generated; gitignored. Contains API tokens, passwords, and auto-generated secrets.
~/.stackr/state.json Deploy state — tracks the rendered compose hash and image digests for every deployed app. Used to skip unchanged apps and support rollback.
~/.stackr/compose/<app>/docker-compose.yml Rendered compose files — written by Stackr before each deploy. One directory per app.
~/.stackr/catalog/ User catalog overlay. Populated by stackr catalog update. When present, replaces the built-in catalog entirely. Delete to revert to the bundled catalog.
~/.stackr/web-stdout.log
~/.stackr/web-stderr.log		 Web UI service logs (Linux systemd only). Rotated automatically.
~/.config/systemd/user/stackr-web.service systemd user unit written by stackr service install on Linux.
/opt/appdata/<app>/ Container data volumes — the data_dir from your config. Each app gets its own subdirectory. Created by Stackr on deploy (with sudo if needed).
Permissions — /opt/appdata
/opt/appdata is owned by root on most Linux installs. Stackr creates subdirectories automatically using sudo mkdir and then sudo chowns them to your user, so no manual setup is required as long as passwordless sudo is configured (the default on most homelab distros).

To set it up manually: 
sudo mkdir -p /opt/appdata
sudo chown -R $USER:$USER /opt/appdata
